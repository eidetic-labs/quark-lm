# Current Evidence

This page records the current promoted run and the evidence that justifies it.
Every value in the snapshot table is read from <code>sites/shared/current-state.json</code>,
so the page cannot drift from the machine-tracked state. The current promoted run
is <code>{state.currentRun}</code>.

Two things stay strictly separate here, as everywhere in QuarkLM: what retrieval
memory can *serve* from the admitted corpus, and what the transformer has
*learned* into its weights. Retrieval answering a probe is `memory-served`
evidence; it is not `weight-consolidated` evidence. See
[Language model](./language-model.md) for the three evidence states and
[Build](../build/index.md) for the two paths that produce them.

Older milestone evidence that is no longer part of the current-state table is
preserved in the [Historical evidence archive](./historical-evidence.md). The
complete version-by-version transformer screen log lives in
[Transformer screen history](../build/transformer-screen-history.md).

## What is promoted

The promoted run is the responder track: the deterministic responder, retrieval
memory, the learned answer classifier, and the generative answer decoder. The
from-scratch transformer is **not** promoted. It remains diagnostic evidence,
blocked on `branch_diversity_target` (see [Transformer status](#transformer-status)
below).

The corpus, the curriculum derived from it, and the probes generated from it are
the only inputs that train weights. Documentation is a promotion gate, not a
training input: a run is promoted only after it updates the docs that describe
current state. Nothing on the responder track imports external weights,
tokenizers, embeddings, or training text — see
[Purity boundary](../secure/purity-boundary.md).

## Transformer status

The transformer is the weight-consolidation path, and it has not cleared its
gate. The blocker is named and unchanged: direct-answer snapshots emit
`branch_diversity_target`, which fails when multi-target eval profiles collapse
to too few predicted branch tokens. The model learns to predict one dominant
first token instead of routing each prompt to its own answer.

Reading these together is the point. The selector and the no-candidate generator
reach exact answers across the eval set while the raw greedy transformer stays at
`0/219`. The system can already *serve* every answer through the responder track
while the neural weights have not yet *learned* to route them. That gap is the
reason the evidence states are kept apart.

```text
serve:  question -> retrieval / responder / selector / generator -> exact answer
learn:  question -> raw greedy transformer ----------------------> not yet routed
```

promotion — is recorded in
[Transformer screen history](../build/transformer-screen-history.md). The durable
explanation of how the transformer works is in
[Transformer](../build/transformer.md).

## Retrieval memory is memory, not learning

Retrieval memory builds cards from the closed-world corpus and answers eval
probes exactly, with provenance and without moving any weights. It is reported as
a transformer answer-training *artifact* so it can be audited, but it never
counts as neural promotion.

| Signal | Current value |
| --- | --- |
| Retrieval memory summary | <code>{state.transformerRetrievalMemorySummary}</code> |
| Retrieval memory status | <code>{state.transformerRetrievalMemoryStatus}</code> |
| Latest retrieval memory report | <code>{state.transformerLatestRetrievalMemoryRun}</code> |

Exact retrieval is `memory-served`. It proves the corpus contains the answer. It
does not prove the transformer learned it.

The responder track is reliable and the corpus is fully covered, but a release
that names a language model cannot be cut while the from-scratch transformer
fails `branch_diversity_target`. The honest claim is the narrow one: the corpus
is admitted and served exactly; the neural weights have not yet learned to route
prompts to distinct answers.
